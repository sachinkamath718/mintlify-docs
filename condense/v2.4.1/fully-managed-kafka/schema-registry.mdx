## Overview

Schema Registry provides centralized schema management for Apache Kafka applications. It enables producers and consumers to share versioned schemas, enforce compatibility rules, and safely evolve event structures across distributed applications.

Every Condense deployment includes a fully managed Schema Registry that supports **Apache Avro®**, **JSON Schema**, and **Protocol Buffers (Protobuf)**. Schemas are versioned and stored centrally, allowing applications to evolve while maintaining compatibility based on the configured compatibility policy.

The Schema Registry exposes a REST API for registering, retrieving, validating, and managing schemas. It also provides serializers that integrate with Apache Kafka clients, enabling automatic schema registration and retrieval for supported serialization formats.

Schema Registry is deployed as part of the Condense platform and is accessible through an internal endpoint within the deployed environment.

## Schema Registry Dashboard

The Schema Registry dashboard provides a centralized interface for managing schema subjects and compatibility settings.

The page includes:

- Global compatibility level
- Subject search
- Schema subject list
- Schema details
- Create Schema action

## Global Compatibility Level

The **Global Compatibility Level** defines the default compatibility policy applied to schemas managed by the Schema Registry.

Compatibility policies help ensure schema evolution does not break existing producers or consumers.

Administrators can configure the global compatibility level directly from the Schema Registry dashboard.

## Schema Subjects

Schema subjects represent versioned schema definitions stored within the Schema Registry.

The subject list allows administrators to:

- Browse registered subjects.
- Search for subjects.
- View schema versions.
- Review compatibility settings.
- Inspect schema definitions.

## Create a Schema

<Frame>
  ![Image](/images/image-107.png)
</Frame>

To create a schema:

1. Navigate to **Schema Registry**.
2. Select **Create Schema**.
3. Enter the subject name.
4. Select the schema type.
5. Define the schema.
6. Create the schema.

After creation, the schema is registered under the specified subject.

## Supported Schema Types

Condense supports the following schema formats.

| Schema Type | Description |
| --- | --- |
| Avro | Apache Avro schema definitions. |
| JSON Schema | JSON-based schema validation. |
| Protobuf | Protocol Buffers schema definitions. |

## REST API Examples

```text
# Register a new version of a schema under the subject "Kafka-key"
$ curl -X POST -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"schema": "{\"type\": \"string\"}"}' \
    http://<internal-lb-ip>:8081/subjects/Kafka-key/versions
  {"id":1}

# Register a new version of a schema under the subject "Kafka-value"
$ curl -X POST -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"schema": "{\"type\": \"string\"}"}' \
     http://<internal-lb-ip>:8081/subjects/Kafka-value/versions
  {"id":1}

# List all subjects
$ curl -X GET http://<internal-lb-ip>:8081/subjects
  ["Kafka-value","Kafka-key"]

# List all schema versions registered under the subject "Kafka-value"
$ curl -X GET http://<internal-lb-ip>:8081/subjects/Kafka-value/versions
  [1]

# Fetch a schema by globally unique id 1
$ curl -X GET http://<internal-lb-ip>:8081/schemas/ids/1
  {"schema":"\"string\""}

# Fetch version 1 of the schema registered under subject "Kafka-value"
$ curl -X GET http://<internal-lb-ip>:8081/subjects/Kafka-value/versions/1
  {"subject":"Kafka-value","version":1,"id":1,"schema":"\"string\""}

# Fetch the most recently registered schema under subject "Kafka-value"
$ curl -X GET http://<internal-lb-ip>:8081/subjects/Kafka-value/versions/latest
  {"subject":"Kafka-value","version":1,"id":1,"schema":"\"string\""}

# Delete version 3 of the schema registered under subject "Kafka-value"
$ curl -X DELETE http://<internal-lb-ip>:8081/subjects/Kafka-value/versions/3
  3

# Delete all versions of the schema registered under subject "Kafka-value"
$ curl -X DELETE http://<internal-lb-ip>:8081/subjects/Kafka-value
  [1, 2, 3, 4, 5]

# Check whether a schema has been registered under subject "Kafka-key"
$ curl -X POST -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"schema": "{\"type\": \"string\"}"}' \
    http://<internal-lb-ip>:8081/subjects/Kafka-key
  {"subject":"Kafka-key","version":1,"id":1,"schema":"\"string\""}

# Test compatibility of a schema with the latest schema under subject "Kafka-value"
$ curl -X POST -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"schema": "{\"type\": \"string\"}"}' \
    http://<internal-lb-ip>:8081/compatibility/subjects/Kafka-value/versions/latest
  {"is_compatible":true}

# Get top level config
$ curl -X GET http://<internal-lb-ip>:8081/config
  {"compatibilityLevel":"BACKWARD"}

# Update compatibility requirements globally
$ curl -X PUT -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"compatibility": "NONE"}' \
    http://<internal-lb-ip>:8081/config
  {"compatibility":"NONE"}

# Update compatibility requirements under the subject "Kafka-value"
$ curl -X PUT -H "Content-Type: application/vnd.schemaregistry.v1+json" \
    --data '{"compatibility": "BACKWARD"}' \
    http://<internal-lb-ip>:8081/config/Kafka-value
  {"compatibility":"BACKWARD"}
```
