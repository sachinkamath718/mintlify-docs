## Available Connectors

Condense provides pre-built connectors to integrate Apache Kafka with messaging systems, cloud platforms, databases, storage services, OEM telematics platforms, and utility services.

    <Frame>

    **Connector Type:** Output Connector

    **Description**

    Publishes processed data to Amazon SQS queues for asynchronous, decoupled processing.

    **Brief Description**

    The Amazon SQS Stream Output Connector in Condense allows streaming data from Condense’s Kafka-native pipelines into Amazon Simple Queue Service (SQS) queues in real-time. Amazon SQS is a fully managed message queuing service that enables decoupling and scaling of distributed systems, microservices, and serverless applications.

    This connector bridges Condense’s high-throughput stream processing with SQS’s reliable message delivery infrastructure—making it ideal for asynchronous workflows, system fan-out, and downstream integrations that require queue-based semantics.

    **Links to Relevant Documents**

    [Amazon SQS Developer Guide](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/welcome.html)

    [Amazon SQS SendMessage API Reference](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/APIReference/API_SendMessage.html)

    [AWS IAM & Credential Configuration](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_access-keys.html)

    **Connector Overview**

    The connector takes messages from an internal Condense Kafka-native topic and pushes them as individual messages to the configured SQS queue.

    It supports:

    A. Secure authentication using AWS access and secret keys

    B. Regional queue targeting (e.g., us-east-1, ap-south-1)

    C. Transparent mapping of streaming events into SQS-compliant JSON payloads

    **Core Functionality**

    1. Sends Kafka messages from Condense directly to an SQS queue

    2. Built-in batching, retry, and exponential backoff on failures

    3. Works with Standard Queues (FIFO queue support is roadmap dependent)

    4. Compatible with downstream AWS Lambda, Step Functions, or EC2 services listening on SQS

    **How it Works in Condense**

    1. A Condense Kafka-native topic receives streaming events.

    2. The SQS Output Connector consumes messages from that topic.

    3. Each message is wrapped into a valid SendMessage request to the specified SQS queue.

    4. Condense ensures delivery acknowledgement, retry handling, and error logging.

    5. Message attributes or deduplication parameters (for FIFO) can be extended via transforms.

    **Configurations**

    | **CATEGORY** | **FIELD NAME** | **DESCRIPTION** | **REQUIRED** |
    | :-- | :-- | :-- | :-- |
    | General | Title | Unique name for identifying this SQS connector | Yes |
    | AWS Details | Input Queue Name | The name of the target SQS queue to send messages to | Yes |
    | AWS Details | Region | AWS region where the SQS queue is hosted (e.g., us-west-2) | Yes |
    | Auth | Access Key | AWS access key with permission to send messages to the queue | Yes |
    | Auth | Secret Key | AWS secret key associated with the above access key | Yes |
    | Condense | Topic (Input) | Kafka-native topic in Condense to consume messages from | Yes |

    **Field-by-Field Explanation**

    1. **Title**
       What It Is: Logical name of this connector in the Condense pipeline UI.
       Tip: Use clear identifiers such as sqs-payment-alert-output.
    2. **Input Queue Name**
       What It Is: The exact name of the destination SQS queue.
       Note: Must exist in the configured region prior to use.
       Example: fraud-events, order-processing-queue, device-data-queue
    3. **Region**
       What It Is: The AWS region where your SQS queue resides.
       Example Values:
       A. us-east-1 (N. Virginia)
       B. ap-south-1 (Mumbai)
       C. eu-central-1 (Frankfurt)
    4. **Access Key**
       What It Is: The AWS IAM access key ID used for API authentication.
       Permissions Required:
       A. sqs:SendMessage
       B. sqs:GetQueueUrl (if not hardcoded)
    5. **Secret Key**
       What It Is: Secret key corresponding to the AWS access key.
       Security Note: Always store securely. Do not hardcode or expose in plain text.
    6. **Topic (Input)**
       What It Is: Kafka-native Condense topic from which messages will be read.
       Content: Messages are expected to be JSON or serialized text payloads suitable for SQS transmission.

    **Sample Configuration Example**

    {

    "title": "sqs-log-output-connector",

    "inputQueueName": "application-log-events",

    "region": "ap-south-1",

    "accessKey": "AKIA\*\*\*\*\*\*\*\*\*\*\*",

    "secretKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCY\*\*\*\*\*\*\*\*\*\*",

    "inputTopic": "[logs.realtime.security](http://logs.realtime.security)"

    }

    **Troubleshooting and Common Issues**

    1. **Access Denied**
       Error: 403 AccessDeniedException
       Solution: Ensure IAM user or role has `sqs:SendMessage` permissions for the queue.
    2. **Queue Not Found**
       Error: AWS.SimpleQueueService.NonExistentQueue
       Solution: Verify the queue name and region match the actual queue.
    3. **Throttling or Rate Limits**
       Error: ThrottlingException or HTTP 429
       Solution: Reduce the message publishing rate or increase provisioned limits via AWS Support.
    4. **Message Too Large**
       Limit: Maximum allowed size per message is 256 KB.
       Solution: Use a transform in Condense to trim, compress, or split large payloads.

    **Advanced Topics**

    1. **FIFO Queue Support**
       A. Current version supports Standard Queues. FIFO support (with `MessageGroupId` and deduplication ID) is planned.
       B. To simulate FIFO behavior, use keys and sequence logic in transforms.
    2. **Dead-Letter Queue (DLQ)**
       A. Use AWS Console or CLI to attach a DLQ to your queue for failed messages.
       B. Condense respects AWS retry policies and delivery delay configurations.
    3. **Message Attributes**
       Future versions of the connector may support setting message attributes (e.g., application, severity, deviceId) via custom logic or transform output.
    4. **Security Best Practices**
       A. Avoid long-lived access/secret keys. Prefer IAM Roles or short-term credentials with rotation.
       B. Use encrypted secrets storage and access via Condense secure environment variables.

    **Best Practices**

    1. Use clear, purpose-specific queue names to avoid confusion.

    2. Monitor queue length and age using Amazon CloudWatch for flow diagnostics.

    3. If publishing high-frequency telemetry, use batch compression or sampling upstream.

    4. Combine this connector with Condense logic transforms to enrich messages before queuing.

    5. Test IAM policies using AWS IAM Policy Simulator before applying to production.

    <Frame>
      ![](/images/condense/v2.4.0/deep-dive/connectors/<replace-with-image>.png)
    </Frame>

    **Connector Type:** Input Connector

    **Description**

    Ingests messages from external Kafka topics into Condense for routing, enrichment, and delivery.

    **Brief Description**

    The Apache Kafka Stream Input Connector allows Condense to consume real-time data from any externally managed or self-hosted Apache Kafka cluster. This connector acts as a bridge, ingesting messages from a specified Kafka topic and publishing them into a Kafka-native topic inside Condense, enabling downstream stream processing, transformations, and real-time application logic.

    It supports secure and authenticated Kafka clusters, with configurability for consumer group tracking and fine-grained offset control.

    **Links to Relevant Documentation**

    Apache Kafka Documentation: [https://kafka.apache.org/documentation/](https://kafka.apache.org/documentation/)

    Kafka Security Overview: [https://kafka.apache.org/documentation/#security](https://kafka.apache.org/documentation/#security)

    Kafka Consumer Configuration: [https://kafka.apache.org/documentation/#consumerconfigs](https://kafka.apache.org/documentation/#consumerconfigs)

    **Connector Overview**

    This connector is designed for importing real-time events from an existing Apache Kafka infrastructure into the Condense platform. It supports TLS, SASL authentication mechanisms, consumer group assignment, and high-throughput ingestion into Condense-managed streaming pipelines.

    **Core Functionality**

    1. Connects to one or more Apache Kafka brokers as an external client

    2. Consumes data from a specified topic with full offset tracking

    3. Supports TLS and SASL-based secure connections

    4. Publishes consumed messages into a Kafka-native Condense topic

    5. Designed for real-time, high-volume event ingestion from enterprise Kafka clusters

    **How It Works in Condense**

    1. The connector establishes a consumer connection to the configured Kafka cluster.

    2. It subscribes to the specified topic using the provided consumer group.

    3. Data is consumed and converted into Condense’s internal streaming format.

    4. Messages are published into the defined Output Topic inside Condense for real-time processing or further routing.

    **Configuration**

    | **CATEGORY** | **FIELD NAME** | **DESCRIPTION** | **REQUIRED** |
    | :-- | :-- | :-- | :-- |
    | General | Title | Unique identifier for the connector in Condense | Yes |
    | Kafka | Bootstrap Servers | Comma-separated list of Kafka broker addresses | Yes |
    | Kafka | Topic Name | Kafka topic to consume from | Yes |
    | Kafka | Security Protocol | Protocol used (PLAINTEXT, SSL, SASL\_SSL) | Yes |
    | Kafka | Kafka Username | Username for SASL authentication | Yes |
    | Kafka | Kafka Password | Password for SASL authentication | No |
    | Kafka | Consumer Group | Kafka consumer group ID used for offset tracking | Yes |
    | Condense | Topic (Output) | Internal Kafka-native topic in Condense to publish data to | Yes |

    **Field-by-Field Explanation**

    1. **Title**
       Human-readable identifier within the Condense UI.
       Used to distinguish among multiple connectors connected to different Kafka sources.
    2. **Bootstrap Servers**
       Example: kafka1.example.com:9092,kafka2.example.com:9092
       One or more brokers used to discover the Kafka cluster.
       These should be publicly accessible if the Kafka cluster is outside Condense.
    3. **Topic Name**
       The source Kafka topic to consume from.
       Data from this topic will be streamed into Condense for further processing.
    4. **Security Protocol**
       Options include:
       A. PLAINTEXT: No security, basic TCP
       B. SSL: TLS encryption only
       C. SASL\_SSL: Authentication (SASL/PLAIN, SASL/SCRAM) over SSL
       Choose according to how the external Kafka cluster is configured.
    5. **Kafka Username / Password**
       Applicable when using SASL\_SSL or other authenticated protocols.
       Typically configured with:
       A. sasl.mechanism=PLAIN
       B. security.protocol=SASL\_SSL
       Username is mandatory when authentication is enabled.
       Password is optional if the mechanism is token-based.
    6. **Consumer Group**
       The Kafka consumer group ID used for offset management.
       Enables consistent processing during connector restart or failover.
       Allows multiple connectors to read independently from the same topic.
    7. **Topic (Output)**
       Condense-native Kafka topic to which consumed records are published.
       Enables downstream apps, transforms, or alerts to process the ingested data.

    **Sample Configuration Example**

    {

    "title": "external-kafka-ingest-finance",

    "bootstrapServers": "broker1.kafka.company.com:9093,broker2.kafka.company.com:9093",

    "topicName": "finance.transactions.raw",

    "securityProtocol": "SASL\_SSL",

    "kafkaUsername": "condense\_user",

    "kafkaPassword": "secure\_password\_123",

    "consumerGroup": "condense-finance-cg",

    "outputTopic": "[condense.finance](http://condense.finance).ingest"

    }

    **Troubleshooting and Common Issues**

    1. **Invalid Credentials**
       A. Symptoms: Authentication failure, repeated connection attempts.
       B. Fix: Double-check Kafka Username, Kafka Password, and authentication mechanism. Confirm with Kafka admin.
    2. **Cannot Connect to Broker**
       A. Symptoms: Connection timeout, no leader error.
       B. Fix: Validate firewall access, DNS resolution, and TLS certificates if using SSL.
    3. **Duplicate Records**
       A. Cause: Incorrect or reused consumer group ID in multiple connectors.
       B. Fix: Assign unique consumer group per connector unless you want shared consumption.
    4. **No Records Consumed**
       A. Possible Causes: Wrong topic name, lagging offsets, or consumer group already committed latest offset.
       B. Fix: Reset consumer group offset if needed using Kafka admin tools.

    **Advanced Topics**

    1. **SASL Mechanisms**
       A. Supported mechanisms include PLAIN, SCRAM-SHA-256, SCRAM-SHA-512.
       B. Configure appropriately in Kafka cluster and Condense connector settings.
    2. **TLS Configuration**
       A. Custom CA certificates not yet configurable directly from UI.
       B. Currently supports default JVM truststore for SSL.
    3. **Offset Management**
       A. Fully managed within Condense per consumer group.
       B. Future enhancement includes offset visualization and replay controls.
    4. **Performance Notes**
       A. Consumes data in parallel partitions for high throughput.
       B. Designed to handle backpressure via Kafka-native flow control.

    **Best Practices**

    1. Use strong SASL credentials and encrypted protocols (avoid PLAINTEXT).

    2. Monitor offsets and lags using your existing Kafka monitoring setup.

    3. Isolate ingestion streams using dedicated consumer groups.

    4. Keep connector Title and Output Topic aligned for easy traceability in logs.

    5. Use small, modular topics at source Kafka for easier debugging and rerouting.
  </Accordion>

    **Connector Type:** Output Connector

    **Description**

    Streams processed messages from Condense into external Kafka topics for downstream use.

    **Brief Description**

    The Apache Kafka Stream Output Connector enables real-time delivery of processed events from Condense to an external Kafka cluster. It connects securely to a Kafka broker, serializes records from an internal Condense topic, and produces them to a specified Kafka topic. This connector is suitable for forwarding transformed or enriched data to external systems, microservices, databases, or other stream processors that consume from Apache Kafka.

    **Links to Relevant Documents**

    [Apache Kafka Documentation](https://kafka.apache.org/documentation/)

    [Kafka Producer API Reference](https://kafka.apache.org/documentation/#producerconfigs)

    [Kafka Security Overview](https://kafka.apache.org/documentation/#security)

    **Connector Overview**

    The Kafka Output Connector acts as a Kafka producer from the Condense platform. It streams structured records from an internal Kafka-native topic to an external Kafka topic with full support for encryption, authentication, batching, and partitioning. It is compatible with any standards-compliant Kafka broker (self-managed or cloud-hosted), including MSK, Confluent, Redpanda, Instaclustr, and more.

    **Core Functionality**

    1. Connects to external Apache Kafka brokers over TCP/SSL/SASL.

    2. Serializes records from a Condense topic and publishes them to a Kafka topic.

    3. Supports secure protocols (SSL, SASL\_SSL).

    4. Maintains high throughput using Kafka producer batching and partitioning.

    5. Enables full pipeline closure from ingestion to output in real-time systems.

    **How it Works in Condense**

    1. The connector reads messages from a specified Condense Kafka topic.

    2. Each message is serialized (usually JSON or Avro).

    3. A Kafka producer is instantiated using the provided credentials and security config.

    4. Messages are published to the external Kafka topic with delivery guarantees (acks).

    **Configurations**

    | **CATEGORY** | **FIELD NAME** | **DESCRIPTION** | **REQUIRED** |
    | :-- | :-- | :-- | :-- |
    | General | Title | Unique name to identify this Kafka connector instance in UI | Yes |
    | Kafka | Bootstrap Servers | Comma-separated list of Kafka brokers | Yes |
    | Kafka | Topic Name | Destination Kafka topic to produce to | Yes |
    | Kafka | Security Protocol | PLAINTEXT, SSL, or SASL\_SSL | Yes |
    | Kafka | Username | Username used for SASL authentication | Yes |
    | Kafka | Password | Password or token used for authentication | No |
    | Condense | Topic (Input) | Internal topic inside Condense to consume from | Yes |

    **Field-by-Field Explanation**

    1. **Title**
       This is the internal name for the connector inside the Condense UI.
       Helps identify output endpoints and trace delivery issues in logs.
    2. **Bootstrap Servers**
       Format: broker1:9092,broker2:9092
       Used to connect to the external Kafka cluster.
       Should include reachable hostnames or IPs with open ports.
    3. **Topic Name**
       The destination topic in the external Kafka cluster.
       Ensure this topic exists with appropriate partitions and ACLs.
       Data will be continuously pushed to this topic as it's published in Condense.
    4. **Security Protocol**
       Specifies how the connection to Kafka is secured:
       **PLAINTEXT**: No encryption or authentication
       **SSL**: TLS-encrypted connection (certificate trust required)
       **SASL\_SSL**: Encrypted connection with SASL authentication
    5. **Kafka Username**
       Applicable when using SASL\_SSL.
       Provided by your Kafka admin or IAM system if you're using Confluent/MSK.
    6. **Kafka Password**
       Used along with the username for authenticating SASL sessions.
       Can be a static password or a token, depending on your Kafka provider.
       Optional if your security protocol is PLAINTEXT or SSL.
    7. **Topic (Input)**
       This is the Condense-native topic that carries processed or enriched data.
       Messages from this topic are serialized and pushed to the external Kafka broker.

    **Sample Configuration Example**

    {

    "title": "push-to-msk-topic",

    "bootstrapServers": "[b-1.msk.prod:9094](http://b-1.msk.prod:9094),[b-2.msk.prod:9094](http://b-2.msk.prod:9094)",

    "topicName": "analytics-pipeline-output",

    "securityProtocol": "SASL\_SSL",

    "kafkaUsername": "condense\_service",

    "kafkaPassword": "secure\_token\_xyz",

    "inputTopic": "[condense.analytics](http://condense.analytics).normalized"

    }

    **Troubleshooting and Common Issues**

    1. **Authentication Errors**
       Symptoms: SASL authentication failed, Connection closed
       Fix: Check Kafka Username, Kafka Password, and mechanism. Ensure SASL is correctly configured on the broker.
    2. **Broker Unreachable**
       Symptoms: Timeout connecting to broker, UnknownHostException
       Fix: Validate network route, firewall, DNS settings. Try using IPs to rule out DNS issues.
    3. **Topic Not Found**
       Symptoms: UnknownTopicOrPartitionException
       Fix: Ensure the Kafka topic exists. If using Confluent Cloud or MSK, confirm correct region and cluster.
    4. **Throttled Delivery or Latency**
       Cause: Large batch sizes or lack of acks.
       Fix: Optimize batch size and enable retries with backoff in Kafka producer configuration.

    **Advanced Topics**

    1. **Kafka Producer Internals**
       A. Uses asynchronous batching to improve throughput.
       B. Acknowledgment setting is acks=all by default for stronger delivery guarantees.
       C. Optional future enhancement: partitioning key assignment for ordering.
    2. **Multi-region Kafka**
       A. When targeting remote Kafka clusters, account for network delay and encryption overhead.
       B. Consider using compression (snappy, lz4) to reduce outbound payload.
    3. **Format and Schema**
       A. Data is serialized in the format used inside Condense (JSON or Avro).
       B. Schema Registry integration (e.g., Confluent Schema Registry) is not required but can be added in custom transforms.

    **Best Practices**

    1. Always use SASL\_SSL for production connections.

    2. Use service accounts with scoped permissions for Kafka access.

    3. Keep connector logs monitored to detect delivery failures or retries.

    4. Separate each use case with different Topic (Input) and Kafka topic names for traceability.

    5. Validate Kafka ACLs and quotas before deploying high-throughput connectors.
  </Accordion>

    **Connector Type:** Output Connector

    **Description**

    Stores processed data from Condense into Azure Blob containers for secure, long-term retention.

    **Brief Description**

    The Azure Blob Storage Store Output Connector allows seamless integration of real-time event streams from Condense into Azure Blob Storage. It supports secure, scalable, and structured streaming of data into cloud-based object storage buckets, ideal for archival, batch analytics, data lake pipelines, compliance snapshots, and downstream ML workflows.

    This connector consumes data from a Kafka-native topic and writes each message into a blob file format (typically JSON, newline-delimited JSON, or CSV) in the specified Azure container.

    **Links to Relevant Documents**

    Azure Blob Storage Overview: [https://learn.microsoft.com/en-us/azure/storage/blobs/storage-blobs-introduction](https://learn.microsoft.com/en-us/azure/storage/blobs/storage-blobs-introduction)

    Azure Blob Authentication & Access: [https://learn.microsoft.com/en-us/azure/storage/common/storage-auth](https://learn.microsoft.com/en-us/azure/storage/common/storage-auth)

    Azure Storage REST API Reference: [https://learn.microsoft.com/en-us/rest/api/storageservices/](https://learn.microsoft.com/en-us/rest/api/storageservices/)

    SAS vs. Shared Key Authentication: [https://learn.microsoft.com/en-us/azure/storage/common/storage-auth-aad-sas](https://learn.microsoft.com/en-us/azure/storage/common/storage-auth-aad-sas)

    Blob Lifecycle Management & Performance: [https://learn.microsoft.com/en-us/azure/storage/blobs/lifecycle-management-overview](https://learn.microsoft.com/en-us/azure/storage/blobs/lifecycle-management-overview)

    **Connector Overview**

    This output connector enables event streaming into Azure Blob Storage, using structured Kafka messages published in Condense. It is ideal for building:

    A. Cost-effective long-term archives

    B. Time-partitioned batch analytics inputs

    C. Cold data lakes that integrate with Synapse, Databricks, or Power BI

    D. Compliance snapshots of streaming data

    Data is written as individual blob objects, commonly grouped by time window or custom partitioning logic defined in the Condense transform layer.

    **Core Functionality**

    1. Event-to-Blob Streaming: Converts Condense events into blob objects in Azure.

    2. Credentialed Access: Supports Shared Key and SAS Token for secure writes.

    3. Flexible Partitioning: Files can be partitioned by time, topic, or schema attributes.

    4. Format Compatibility: Supports JSON, NDJSON, and custom transform output formats.

    5. Reliable Delivery: Ensures write acknowledgments and error visibility via logs.

    **How It Works in Condense**

    1. Connector subscribes to a Kafka-native topic in Condense.

    2. Each message is streamed in real-time from Condense to the output connector.

    3. The connector authenticates with Azure using either Shared Key or SAS Token.

    4. Messages are converted to file format and written as blob objects into the configured container.

    5. Files are created based on internal batching logic (e.g., time window, byte size, or record count).

    **Configuration**

    | **CATEGORY** | **FIELD NAME** | **DESCRIPTION** | **REQUIRED** |
    | :-- | :-- | :-- | :-- |
    | General | Title | Unique name to identify the connector within Condense | Yes |
    | Storage | Storage Account Name | Name of the Azure storage account (without .[blob.core.windows.net](http://blob.core.windows.net)) | Yes |
    | Storage | Storage Container Name | Container (bucket) name inside Azure Blob Storage | Yes |
    | Storage | Storage Access Key | Key for accessing Azure storage (Shared Key or SAS Token) | Yes |
    | Storage | Storage Credential Type | Type of credential: shared\_key or sas\_token | Yes |
    | Condense | Topic (Input) | Kafka-native Condense topic to consume streaming events from | Yes |

    **Field-by-Field Explanation**

    1. **Title**
       Purpose: Internal reference name for the connector within Condense.
       Example: azure-archive-2025, telemetry-backup, audit-trail-writes
    2. **Storage Account Name**
       Definition: Name of the Azure storage account to which blobs are written.
       Notes:
       A. Exclude protocol prefix or blob suffix.
       B. Must be globally unique in Azure.
       Example: zeliotdatastore, iottelemetrydata
    3. **Storage Container Name**
       Definition: Name of the blob container inside the Azure storage account.
       Constraints:
       A. Must be lowercase.
       B. Can be created in advance or auto-created by Condense.
       Example: event-streams, archive2025, realtime-dumps
    4. **Storage Access Key**
       Credential used to authenticate with Azure Blob Storage.
       Types:
       A. Shared Key: Base64 encoded primary or secondary access key.
       B. SAS Token: Pre-generated URL-safe token with scoped permissions.
       Security Tips:
       A. Use limited-scope SAS token for better security.
       B. Rotate keys periodically.
       Where to get:
       A. Azure Portal → Storage Account → Access Keys
       B. Or generate SAS Token from “Shared Access Signature” tab
    5. **Storage Credential Type**
       Definition: Determines how Condense connects to Azure Blob Storage.
       Accepted Values:
       A. shared\_key: Uses full access key.
       B. sas\_token: Uses time-bound URL token (recommended for production).
       Example: shared\_key or sas\_token
    6. **Topic (Input)**
       Definition: Kafka-native Condense topic to consume records from.
       Requirement: Messages must be well-structured (JSON, CSV, etc.).
       Tip: Apply preprocessing transform for field selection, file formatting, or schema alignment before sending to this connector.

    **Sample Configuration Example**

    {

    "title": "azure-lake-writer",

    "storageAccountName": "iotanalyticsprod",

    "storageContainerName": "vehicle-trips",

    "storageAccessKey": "REDACTED",

    "storageCredentialType": "shared\_key",

    "inputTopic": "trip-summary-events"

    }

    **Troubleshooting and Common Issues**

    1. **Unauthorized Access / 403 Errors**
       A. Cause: Invalid or expired credential.
       B. Solution: Revalidate SAS token or Shared Key. Ensure time window validity.
    2. **Blob Not Created**
       A. Cause: Improper topic mapping or empty payloads.
       B. Solution: Confirm message structure and use a logging transform for inspection.
    3. **Container Not Found**
       A. Cause: Container does not exist and user lacks permission to auto-create.
       B. Solution: Manually create the container or elevate connector permissions.
    4. **Write Latency**
       If you observe lag or slow uploads:
       A. Confirm network latency to Azure Blob endpoint.
       B. Use larger batch sizes or optimize downstream transform performance.

    **Advanced Considerations**

    1. **File Format & Naming Strategy**
       A. Condense uses time-windowed file batching (e.g., per 5 minutes or 1000 records).
       B. File names typically include timestamp, topic name, and partition offset metadata.
       C. Custom transform logic can modify payload or generate key/value file blobs.
    2. **Access Tier**
       A. Files written can be assigned to Hot, Cool, or Archive tiers post-write using Azure lifecycle rules.
    3. **Storage Class & Region**
       A. Condense does not manage replication—choose LRS/GRS redundancy per storage account.

    **Best Practices**

    1. Use SAS Token with read/write scoped access and expiry.

    2. Organize blob containers by use case or data retention needs.

    3. Integrate with Azure Data Factory or Synapse for downstream ETL.

    4. Apply schema validation or transformation in Condense before blob write.

    5. Set up lifecycle management rules in Azure to auto-expire aged data.
  </Accordion>

    **Connector Type:** Output Connector

    **Description**

    Stores processed real-time data into BigQuery tables for query, dashboard, and analysis.

    **Brief Description**

    The BigQuery Store Output Connector enables real-time event ingestion from Condense Kafka-native topics directly into Google BigQuery tables. This allows users to stream data into Google’s cloud-native data warehouse for immediate querying, analytics, dashboards, compliance tracking, or ML feature extraction.

    **How It Works**

    Once configured:

    1. The connector subscribes to the specified Kafka topic inside Condense.

    2. For every message on that topic, the connector transforms the payload (if required) and uses the AWS Kinesis PutRecord or PutRecords API to publish to the target stream.

    3. Messages are batched and retried as per internal buffer settings to optimize throughput and minimize costs.

    **Stream Behavior and Batch Delivery**

    1. Batching: Condense automatically batches Kafka records and uses AWS's PutRecords API to reduce API throttling and improve delivery efficiency.

    2. Partition Keys: Currently, Condense uses a default hash-based partition key for message distribution. Custom partition key support is in the roadmap.

    3. Retries: In case of transient AWS errors (throttling, network), retries are applied using exponential backoff up to a maximum retry threshold.

    4. Ordering Guarantees: Kinesis ordering is per shard, not global. If strict ordering is required, users should plan topic-to-shard mapping accordingly.

    **Security and Authentication**

    1. **IAM Role Best Practices**
       It is recommended to:
       A. Use a dedicated IAM user or role for Condense with the minimum required permissions: kinesis:PutRecords, kinesis:DescribeStream, and optionally cloudwatch:PutMetricData for observability.
       B. Rotate Access Keys periodically.
       C. Use environment variables or secret vault integrations to securely manage access credentials.

    **Error Handling and Logging**

    Condense logs all delivery attempts:

    1. Success: Logged with message size, latency, and Kinesis shard ID.

    2. Failure: Logged with AWS error code (e.g., ProvisionedThroughputExceededException) and retried if transient.

    3. For non-retriable errors, alerts are raised and messages are moved to a dead-letter topic if configured.

    **Monitoring and Metrics**

    Metrics tracked:

    A. Message delivery latency

    B. Batch size and throughput (records/sec)

    C. Failure count and failure reasons

    D. AWS SDK-level response codes

    Integration with external monitoring tools like CloudWatch, Prometheus, or OpenTelemetry is possible using Condense native observability modules.

    **Advanced Considerations**

    1. **Performance Tuning**
       A. Shard Count: Ensure that the destination Kinesis stream has enough shards to match expected write throughput (1 MB/s or 1000 records/sec per shard).
       B. Kafka Message Size: Kinesis has a max record size of 1 MB. Split oversized messages or compress prior to publish.
       C. Latency Optimization: Keep batches small if low latency is preferred; larger batches yield better cost-performance ratios.
    2. **Region Selection**
       Choose the AWS region that:
       A. Matches the consumer region (e.g., AWS Lambda, Firehose).
       B. Complies with data residency or regulatory requirements.

    - Condense automatically unpacks and re-streams them as independent events.

    - Label topics based on use case (driver\_alerts, ignition\_events, overspeed\_logs). 

    - Keep SASL credentials secure and rotate passwords periodically. 

    - Monitor parsing metrics and log volume under **Connector Status** for early diagnostics.
